Safety gaps

  • No dependable enforcement of helmet wear before ride start.
  • Accidents can go undetected if riders cannot use the phone.
  • Emergency contacts receive delayed or incomplete context.
  • Unsafe riding behavior is rarely surfaced early.

Security and operations gaps

  • Stolen bikes and helmets lack immediate disable pathways.
  • Fleet operators cannot monitor compliance in one place.
  • Connection loss creates silent failure states for users.
  • Existing ignition systems provide no traceable override logic.
02 · Research

What riders needed most.

Interviews and journey notes pointed to one priority: safety has to react instantly, while routine interactions remain calm.

1. Safety Assessment

Mapped rider journeys and identified where helmet compliance and emergency response fail under time pressure.

2. Research

Interviews and journey notes confirmed that riders need fewer prompts and stronger automatic safety checks.

3. Flow Development

Built a clear sequence from helmet detection to ignition unlock, including critical failure and alert states.

4. Design Direction

Translated insights into a minimal in-ride interface with clear safety status, alerts, and controlled overrides.

04 · Solution

What we built in v1 and what comes next.

The shipped foundation enforces compliance and emergency response first, with analytics and deeper controls planned for later phases.

Must-wear ignition lock

  • Present · v1.0 safety gate is active by default.
  • Ignition stays locked until helmet wear and pairing are both valid.
  • Rider sees clear status feedback before ride start.

Crash detection with auto-alert

  • Present · v1.0 impact workflow triggers automatically.
  • Emergency contacts receive location details without rider input.
  • Alert context is structured for faster response decisions.

Emergency button and timed disable

  • Present · v1.0 physical trigger is available for risk moments.
  • Ignition can be disabled immediately in unsafe situations.
  • Restore actions require verified app authentication.

Authenticated re-enable controls

  • Present · v1.0 override recovery is time-bounded.
  • Each re-enable action requires identity verification.
  • Event logs keep recovery flow auditable and traceable.

Live GPS analytics dashboard

  • Future · phase 2 adds route and behavior analytics.
  • Guardians get clearer visibility into ride safety patterns.
  • Fleet operators can review trend-based risk signals.

Fleet-level remote ignition control

  • Future · phase 3 introduces verified remote lock controls.
  • Lost helmet and theft workflows get stricter policy checks.
  • Support teams can respond faster with safer safeguards.

06 · Delivery principles

What stayed non-negotiable through implementation.

Safety gates are state-based, not reminder-based

Core enforcement lives in device state and backend logic, not user memory.

Critical alerts remain visible until acknowledged

Theft, disconnect, and SOS events persist with traceable status changes.

Overrides are temporary and auditable

Every exception path has expiration, user verification, and event logs.

07 · Reflection

Risks, learnings, and next iteration focus.

The concept validated quickly, while hardware reliability and emergency edge cases became the highest-priority constraints.

Core focus: trust in edge cases

What worked

  • Helmet-to-ignition concept was immediately understood by riders.
  • Guardian alert model created strong trust in early feedback.
  • Control center structure kept feature discoverability high.

Current risks

  • Sensor calibration drift can create false lock and unlock states.
  • Network instability affects alert delivery confidence.
  • Power draw from GPS and sensors needs tighter optimization.

Next steps

  • Run moderated commuting tests in live urban routes.
  • Refine emergency copy and confirm-response controls.
  • Prototype fleet dashboard with real event streams.
